# 比亚迪车机第三方桌面"设为默认桌面"方案调研

**调研日期**: 2026-07-01
**目的**: 调研竞品第三方桌面如何实现默认桌面，评估各技术方案在 BYD DiLink 系统上的可行性

---

## 一、竞品实际做法

| 竞品 | 方案 | 技术实现 |
|------|------|----------|
| **嘟嘟桌面 PRO** | 设置页引导 + 系统选择器 | 应用内"设置 → 系统 → 默认桌面"引导用户到系统原生设置页 |
| **布丁UI** | 首次安装引导弹窗 | 首次启动弹出"设为默认桌面"引导，失败后可在设置中重新触发 |
| **BricksOpenLauncher** | 动态 stub Activity（**唯一有源码的方案**） | 默认禁用的 `HomeStubActivity` 声明 `CATEGORY_HOME`，运行时动态启用触发系统选择器 |
| **极流桌面** | HOME intent-filter + 开机自启 | Manifest 声明 `HOME/DEFAULT/MONKEY`，加 `BootReceiver` 开机拉起 |
| **AutoHub** | HOME intent-filter（不完整） | 只声明了 `HOME` 没声明 `DEFAULT`，部分车型可能不被识别 |
| **byd-split-launcher** | 开机自启覆盖（不替换桌面） | `BootReceiver` + 延迟 8 秒启动分屏，不声明 HOME |

---

## 二、可用技术方案（按可行性排序）

### 方案 1：标准 HOME Intent Filter + 系统选择器

```xml
<intent-filter>
    <action android:name="android.intent.action.MAIN" />
    <category android:name="android.intent.category.HOME" />
    <category android:name="android.intent.category.DEFAULT" />
</intent-filter>
```

- **原理**：安装后按 HOME 键，系统弹出"选择桌面"对话框，用户选择"始终"
- **BYD 验证**：嘟嘟桌面、布丁UI、极流桌面均已验证可行
- **限制**：必须用户手动确认；部分车型重启后可能被系统重置
- **⚠️ BYD 实测问题**：在比亚迪 DiLink 系统上，Manifest 中声明 `CATEGORY_HOME` 会导致 APK 无法安装，车机提示"无法安装桌面类应用"。BYD 在安装时扫描 Manifest intent-filter，检测到 HOME 类别直接拒绝——这是厂商层面的白名单机制

### 方案 2：动态 Stub Activity 触发选择器

BricksOpenLauncher 的方案，核心思路：

```kotlin
// 创建一个默认 disabled 的 HomeStubActivity（声明 CATEGORY_HOME）
// 运行时动态启用它 → 系统发现新 HOME 候选者 → 弹出选择器

fun enableDefaultLauncher() {
    packageManager.setComponentEnabledSetting(
        ComponentName(this, HomeStubActivity::class.java),
        PackageManager.COMPONENT_ENABLED_STATE_ENABLED,
        PackageManager.DONT_KILL_APP
    )
    // 触发系统重新评估
    startActivity(Intent(Intent.ACTION_MAIN).addCategory(Intent.CATEGORY_HOME))
}
```

- **优势**：可以在应用内通过按钮触发，用户体验更好
- **限制**：仍需用户手动选"始终"
- **⚠️ BYD 待验证**：`android:enabled="false"` 的 Activity 是否能绕过 BYD 的安装检测。BYD 的检测逻辑可能只看 Manifest 声明而不管 `enabled` 属性

### 方案 3：开机自启 BOOT_COMPLETED

```xml
<receiver android:name=".BootReceiver">
    <intent-filter>
        <action android:name="android.intent.action.BOOT_COMPLETED" />
    </intent-filter>
</receiver>
```

- **BYD 验证**：byd-split-launcher 用延迟 8 秒启动，已验证可行
- **限制**：不是真正的默认桌面，按 HOME 键仍回原车桌面；开机会短暂闪过原车桌面

### 方案 4：ADB `pm set-home-activity`（迪UI 已具备 ADB 能力）

```bash
adb shell pm set-home-activity com.diui.launcher/.MainActivity
# 或
adb shell cmd package set-home-activity com.diui.launcher/.MainActivity
```

- **优势**：静默设置，不需要用户交互确认
- **BYD 可行性**：NIO 车机已验证，BYD 同理（DiLink Android 9）
- **迪UI 独特优势**：项目已有完整的本地 ADB 客户端（`AdbHelper`），可以直接通过 `execShell` 执行此命令，无需外部工具
- **待验证**：`pm set-home-activity` 和 `cmd package set-home-activity` 在 Android 9 上是否存在，不同 Android 版本可用命令不同

### 方案 5：ADB 停用原车桌面（激进方案）

```bash
adb shell pm disable-user --user 0 com.byd.launcher
```

- **BYD 验证**：汽车之家论坛大量用户验证，甲壳虫 ADB 助手可操作
- **恢复命令**：`adb shell pm enable --user 0 com.byd.launcher`
- **⚠️ 风险高**：如果第三方桌面崩溃或被卸载，车机变砖

### 方案 6：DevicePolicyManager / Accessibility Service

- **DevicePolicyManager**：需要设为 Device Owner，操作复杂，**不推荐**
- **Accessibility Service**：BYDMate 验证 BYD 支持无障碍服务，但 HOME 键是系统特殊键，无障碍服务拦截不到，只能做辅助功能

## 三、BYD 系统限制

- **安装检测机制**：DiLink 系统在安装 APK 时扫描 Manifest，检测到 `CATEGORY_HOME` 直接拒绝安装，提示"无法安装桌面类应用"。这是厂商白名单机制
- **DiLink 3.0/4.0（Android 9）**：安装限制存在，但 ADB 方式可绕过
- **DiLink 5.0+**：可能有更多限制，但 ADB 方式仍可操作
- **约 30 天清除权限**：BYD 会定期清除第三方应用权限，可能也会重置默认桌面设置，需配合开机自启方案

---

## 四、迪UI 推荐策略

迪UI 相比其他竞品有一个**独特优势**：已实现完整的本地 ADB 协议客户端（`AdbHelper`），可以在应用内直接执行 shell 命令。

### 推荐组合方案

| 层级 | 方案 | 作用 |
|------|------|------|
| **基础** | Manifest 中**不声明** `CATEGORY_HOME` | 确保能正常安装到 BYD 车机 |
| **自动** | ADB `pm set-home-activity`（利用现有 AdbHelper） | 真车上静默设为默认，无需用户交互 |
| **兜底** | 开机自启 `BOOT_COMPLETED` | 即使未设为默认也能覆盖显示 |
| **高级** | 设置页提供 ADB 命令指引 | 给有甲壳虫等工具的用户 |

### 关键待验证项

1. `pm set-home-activity` 命令在 Android 9（DiLink 4.0）上是否存在
2. `cmd package set-home-activity` 命令在 Android 9 上是否可用
3. 如果两者都不可用，是否有其他 ADB 方式可以设置默认桌面
4. BricksOpenLauncher 的 `enabled="false"` stub Activity 方案能否绕过 BYD 安装检测
