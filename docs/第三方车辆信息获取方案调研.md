# 第三方比亚迪车辆信息获取方案调研

**调研日期**: 2026-07-02
**目标**: 调研 GitHub/论坛等渠道中，第三方比亚迪软件在获取 ADB 后获取车辆信息的方式，寻找扩展 FID 的方法
**背景**: 当前项目通过 FID 读取 autoservice，但已知 FID 数量不足，无法满足桌面软件需求

---

## 一、信息获取途径总览

通过 ADB 获取比亚迪车辆信息，目前社区中主要有 **6 种途径**：

| 途径 | 难度 | 数据量 | 是否需要额外硬件 | 适用场景 |
|------|------|--------|-----------------|---------|
| autoservice FID 调用 | 中 | 最大 | 否 | 桌面软件（当前方案） |
| **Diplus 本地 HTTP API** | **低** | **大** | **否** | **桌面软件（推荐补充方案）** |
| 官方 SDK 253 接口 | 低 | 大 | 否 | 车机 Android 应用 |
| 逆向工程 autoservice | 高 | 大 | 否 | 扩展 FID 列表 |
| CAN 总线直接读取 | 高 | 最大 | 是（CANable/ELM327） | 底层数据获取 |
| 云端 MQTT API | 中 | 中 | 否（需网络） | 远程车辆状态 |

## 二、途径详解

### 2.1 autoservice 系统服务（当前方案）

autoservice 是 BYD 车机内置的系统 Binder 服务，原生系统通过它与车辆硬件通信。当前项目正是通过 `service call autoservice` 命令读写 FID。

**社区中使用 autoservice 的项目：**

#### BYDMate
- **仓库**: https://github.com/AndyShaman/BYDMate
- **定位**: DiLink 5.0 Android 应用（方程豹豹3 / Leopard 3 实测）
- **核心**: 直接通过 autoservice 系统服务读取数据和控制车辆，通过无线 ADB 的 shell 权限访问
- **可读数据**: BMS 电池健康度(SoH)、SOC、电压、温度、充电功率、充电日志
- **可控功能**: 车窗、空调、灯光、门锁、天窗、座椅加热/通风、前备箱
- **价值**: 是目前 FID 映射最完整的开源项目，项目中的 `FidMap.kt` 文件已被我们提取并整理到 `autoservice-FID参考手册.md`

#### byd-apps
- **仓库**: https://github.com/wheregoes/byd-apps
- **定位**: 开源 BYD 电动车 Android 应用集合，不需要 root
- **核心**: 通过逆向工程的 BYD 内部 Android 服务直接与 CAN 总线交互
- **关键文件**: `research/byd-auto-api-reference.md` — 包含 AC、门锁、全景、车身、云通信等 API 的**实车验证值**
- **固件**: 在 13.1.32.2507250.1 上测试通过
- **价值**: API Reference 文档中可能包含我们尚未发现的 FID 映射

#### byd-trip-stats
- **仓库**: https://github.com/angoikon/byd-trip-stats
- **定位**: BYD DiLink 车载分析仪表盘
- **核心**: 使用 BYD SDK，支持直接轮询车辆数据，带回退监听器
- **覆盖设备**: charging、statistic、climate、instrument、speed、location、energy
- **价值**: 验证了哪些 Device 类在实车上可用

### 2.2 Diplus（迪加）本地 HTTP API（推荐补充方案）

> **车上已安装 Diplus，可直接使用此方案扩充车辆数据**

Diplus（迪+/迪加）是比亚迪车机上的第三方应用，提供千里眼、哨兵模式、车况数据监控等功能。它内部封装了 BYD autoservice 的调用，并对外暴露了本地 HTTP API，端口 **8988**。

#### 核心原理

```
桌面软件 → ADB 端口转发(8988) → Diplus HTTP API → autoservice → CAN 总线
```

Diplus 在应用层封装了 BYD SDK 调用，将底层 FID 转换为中文传感器名称暴露。相当于 **Diplus 帮你做了反编译的工作**——你不需要知道 FID 值，只需用中文名查询即可。

#### API 接口

**1. 单传感器查询 `getVal`**

```
GET http://<车机IP>:8988/api/getVal?name=<中文传感器名>&status=true
```

响应格式：
```json
{"val": 67.5}
```

示例：
```bash
# 查询电量百分比
curl "http://192.168.10.10:8988/api/getVal?name=电量百分比&status=true"
# 返回: {"val": 67.5}

# 查询车速
curl "http://192.168.10.10:8988/api/getVal?name=车速&status=true"
# 返回: {"val": 0}
```

**2. 批量查询 `getDiPars`**

```
GET http://<车机IP>:8988/api/getDiPars
```

一次性返回所有已注册传感器的当前值，[byd-hass](https://github.com/jkaberg/byd-hass) 项目每 8 秒调用一次此接口。

**3. 查询格式（byd-hass 内部实现）**

byd-hass 源码中的查询模板格式为：
```
<GoFieldName>:{<中文传感器名>}
```
多个传感器用管道符 `|` 分隔，通过 URL 参数 `text` 传递：
```
BatteryPercentage:{电量百分比}|Speed:{车速}|Mileage:{里程}
```

返回的原始值需乘以缩放因子（ScaleFactor）才是实际值：
- 里程：`ScaleFactor=0.1`（原始值 × 0.1 = km）
- 胎压：`ScaleFactor=0.01`（原始值 × 0.01 = bar）

#### 已知传感器名称列表

以下传感器来自 byd-hass 项目源码和社区验证，使用中文名称作为 `name` 参数：

| 中文名称 | 含义 | 缩放因子 | 单位 | 对应 FID 状态 |
|---------|------|---------|------|--------------|
| `电量百分比` | 电池 SOC | 1 | % | = FID_SOC |
| `车速` | 当前车速 | 1 | km/h | = FID_SPEED |

#### 在桌面软件中集成的实现方案

**方式 A：ADB 端口转发（推荐）**

当前项目已通过 ADB 连接车机，可利用 ADB 端口转发访问 Diplus API：

```bash
# 通过 ADB 转发端口
adb forward tcp:8988 tcp:8988
# 然后本地访问
curl "http://127.0.0.1:8988/api/getVal?name=电量百分比&status=true"
```

在项目中，可以在 `AdbHelper.java` 中添加端口转发逻辑，然后创建 `DiplusClient.java` 调用 HTTP API。

**方式 B：ADB shell + curl/wget**

如果端口转发不可用，可通过 ADB shell 在车机内部调用：

```bash
adb shell "curl -s 'http://localhost:8988/api/getVal?name=电量百分比&status=true'"
```

**实现架构建议：**

```
BydVehicleManager.readCurrentStatus()
  ├── BydAcApi / BydBodyworkApi ...  (BYD SDK 反射，signature 权限可能被拒)
  ├── fillFromAutoserviceShell()     (FID 方案，当前主力)
  ├── fillExtrasFromHelper()         (HelperDaemon FID 方案)
  └── fillFromDiplus()               (⭐ 新增：Diplus HTTP API 补充数据)
```

`fillFromDiplus()` 的定位是：**为 FID 无法获取的字段提供数据补充**，特别是：
- 油量百分比、续航里程、纯电续航里程
- 瞬时电耗/油耗、平均电耗/油耗
- 发动机转速、电机转速、方向盘角度

1. Diplus 必须在车机上**运行中**才能响应 API 请求
2. 部分新版 Diplus 可能调整了 API 路径或添加了认证

4. Diplus 本身也是通过 BYD SDK/autoservice 获取数据，所以数据来源与 FID 相同，但它内部已经做了 FID 映射

### 2.3 CAN 总线直接读取

通过 OBD 接口或车机内核 SocketCAN 直接读取底层 CAN 报文。

#### 已知可获取数据（比亚迪 E3 等车型验证）

- 车门状态、引擎盖、车灯（远近光/转向/双闪）
- 雨刮开关、车门锁、运动模式、钥匙状态
- 刹车踏板/加速踏板幅度、方向盘转向角度
- 档位信号、驻车手刹、总里程、车架号、车速
- 安全带状态、空调温度及风量

#### 比亚迪 CAN 总线特殊性

- 比亚迪完全拥有自主的 CAN 矩阵设计
- OBD 诊断仪都是自研的
- CAN 针脚与标准燃油车不同：
  - 部分车型使用 OBD 第6脚/第14脚（HS-CAN/MS-CAN）
  - 早期车型（如 E5）使用第3脚/第11脚（动力电池数据）+ 第12脚/第13脚

#### 相关项目

| 项目 | 说明 |
|------|------|
| [opendbc-byd](https://github.com/BYDcar/opendbc-byd) | BYD 专属 CAN DBC 信号数据库（CAN 信号定义文件） |
| [BYD-PID-list](https://github.com/loryanstrant/BYD-PID-list) | 比亚迪 OBD PID 列表，Dolphin/Atto 3/Seal 可能通用 |
| [commaai/opendbc](https://github.com/commaai/opendbc) | 上游 opendbc 项目（Python API） |

#### 车机内核 SocketCAN 方式（如果支持）

```bash
# 通过 ADB Shell 配置 CAN 接口
adb shell su -c "ip link set can0 down"
adb shell su -c "ip link set can0 type can bitrate 500000"
adb shell su -c "ip link set can0 up"

# 抓取 CAN 数据
adb shell su -c "candump can0"      # 打印所有数据包
adb shell su -c "cansniffer can0"   # 只显示变化的报文
```

> **注意**: 需要 root 权限 + 车机内核 SocketCAN 支持，实际可行性取决于车型和系统版本

### 2.4 比亚迪云端 MQTT API

**代表项目**: [Niek/BYD-re](https://github.com/Niek/BYD-re)

- 逆向了比亚迪的云端通信协议

## 三、扩展 FID 的具体方案

### 方案 A：反编译 framework.jar 提取 FID（推荐，优先级最高）

**原理**: BYD SDK 的每个 getter 方法内部都会调用 autoservice 并传入 FID 常量。反编译 framework 就能直接拿到 SDK 方法 → FID 的映射。

**步骤**:
1. 通过 ADB 从车机拉取 BYD 框架：
   ```bash
   adb pull /system/framework/bydauto.jar
   # 或者
   adb pull /system/framework/oat/arm64/bydauto.odex
   ```
2. 用 jadx 或 dex2jar 反编译
3. 在 `BYDAutoStatisticDevice` 等类的每个 getter 方法中提取 FID 常量
4. 整理映射关系，补充到 `FidRegistry.java` 和 `autoservice-FID参考手册.md`

**预期收获**: 可以获取 253 个官方接口对应的全部 FID

### 方案 B：参考 byd-apps API Reference

**步骤**:
1. 克隆 https://github.com/wheregoes/byd-apps
2. 阅读 `research/byd-auto-api-reference.md`
3. 对照已有 FID 列表，找出新的 FID/API 映射
4. 该文档包含实车验证值，可直接信赖

### 方案 C：FID 范围扫描

**原理**: 已知 FID 的数值在一定范围内，通过逐个探测附近值发现新的有效 FID。

**步骤**:
1. 围绕已知 FID（如 `FID_SOC = 1246777400`）的 ±100~±1000 范围扫描
2. 对每个候选值执行 `service call autoservice 5/7 i32 <dev> i32 <candidate_fid>`
3. 过滤掉哨兵值（65535/1048575/-10013/-10011）
4. 记录返回有效数据的 FID

> 当前项目已有「FID 全量扫描」功能，可利用它批量探测

### 方案 D：参考 BYDMate 源码获取更多 FID

**步骤**:
1. 克隆 https://github.com/AndyShaman/BYDMate
2. 查看 `FidMap.kt` 中是否有我们尚未收录的 FID
3. 特别关注写入 FID（已在 FID 参考手册第七节部分收录）
4. BYDMate 持续更新，定期同步新发现的 FID

### 方案 E：注册官方开发者获取完整文档

**步骤**:
1. 注册 https://oip.byd.com/ 开发者账号
2. 实名认证
3. 下载「比亚迪智慧开放平台API说明书V1.0.5.pdf」
4. 说明书中应包含完整的 FID/接口映射

## 六、术语对照

不同项目/社区使用不同术语指代类似概念：

| 术语 | 全称 | 含义 | 使用场景 |
|------|------|------|---------|
| **FID** | Feature ID | autoservice 内部功能标识，对应 CAN 信号地址 | 本项目、BYDMate |
| **PID** | Parameter ID | OBD 标准协议参数标识 | BYD-PID-list、OBD 诊断 |
| **SDK API** | — | 官方 SDK 暴露的 Device 类方法名 | oip.byd.com、CSDN 文章 |
| **CAN Signal** | — | DBC 文件中的信号定义 | opendbc-byd |
| **dev** | Device Type | autoservice 设备类型编号 | 本项目 |
| **tx** | Transaction Code | autoservice 事务码（5=int, 7=float） | 本项目 |
